## Problem Statement

CloudEnv deploys containers to Fly.io, but real-world docker-compose files designed for local dev break on cloud deployment. Bind mounts inject files that don't exist on Fly. Env vars reference unset variables. Init scripts are mounted, not baked in. Services depend on docker.sock. The CLI can detect these issues (preflight check) and skip broken services, but it can't fix them.

The insight: the coding agent (Claude Code) already sees the entire project. It knows what files exist, what the Dockerfiles do, what the compose file expects. It just needs to be taught the cloudenv workflow. Then it can analyze the project, generate fixes, and call cloudenv with everything set up correctly.

## Recommended Approach

**Claude Code Skill** that teaches the agent the cloudenv deployment workflow.

The skill lives at `.claude/skills/cloudenv-deploy/SKILL.md` in any project (or `~/.claude/skills/cloudenv-deploy/` for global install). When a user says "deploy this" or "create a preview environment", the agent:

1. Reads docker-compose.yml (or docker-compose.prod.yml)
2. Identifies issues: bind mounts, missing env vars, docker.sock, dev targets
3. For each issue, generates a fix:
   - Bind mount → generate wrapper Dockerfile that COPYs the files
   - Missing env vars → use defaults or ask the user
   - docker.sock → skip the service
   - Dev target → use prod target if available
4. Calls `cloudenv up` with the fixed config
5. Verifies the URL is reachable
6. Reports the result

The agent does the thinking. Cloudenv does the deploying.

### Skill Flow

```
User: "deploy this to a preview env"
  │
  Agent reads SKILL.md (learns cloudenv workflow)
  │
  ├── Step 1: Read docker-compose*.yml files
  │   Pick prod over dev if both exist
  │
  ├── Step 2: Analyze for cloud-readiness
  │   Scan for bind mounts, env vars, docker.sock
  │   Read referenced files to understand what they do
  │
  ├── Step 3: Generate fixes
  │   For services with bind mounts:
  │     Generate Dockerfile.cloudenv.{service} that:
  │       FROM {original image}
  │       COPY {mounted files} {mount target}
  │     Write to project dir (gitignored)
  │   For env vars: resolve ${VAR:-default} or ask user
  │
  ├── Step 4: Generate cloudenv-ready compose
  │   Write docker-compose.cloudenv.yml with:
  │     - build: pointing to generated Dockerfiles
  │     - env vars resolved
  │     - broken services removed
  │
  ├── Step 5: Deploy
  │   Run: cloudenv up -f docker-compose.cloudenv.yml
  │
  └── Step 6: Verify
      Curl the URL, check response
      Report: "Your preview is live at https://..."
```

### What the CLI Needs to Support

The CLI already supports everything. The skill uses:
- `cloudenv up -f <file>` — deploy a specific compose file
- `cloudenv up --port <port>` — override web port
- `cloudenv up --name <name>` — override env name
- `cloudenv status` — check health
- `cloudenv down` — tear down

No new CLI features needed. The skill orchestrates existing commands.

### What the Skill Needs to Teach

The skill's SKILL.md contains:
1. When to trigger (user says "deploy", "preview", "cloudenv up")
2. How to analyze a compose file for cloud-readiness
3. How to generate wrapper Dockerfiles for bind mounts
4. How to generate a cloudenv-ready compose file
5. How to call cloudenv CLI
6. How to verify the deployment worked

## Future: Internal AI Validation Layer

Phase 2: Add an `--ai-validate` flag to cloudenv that sends the deployment plan to Claude API for validation before executing. This catches issues the static preflight check misses. The skill handles most cases, but standalone CLI users (no agent) benefit from this too.

## Success Criteria

1. A Claude Code user can say "deploy this to a preview env" in any project with a docker-compose.yml and get a working URL
2. InsForge's docker-compose.prod.yml deploys successfully with all services healthy
3. The skill handles bind mount → COPY conversion automatically
4. No changes to cloudenv CLI required
