## Affected Pages/Routes
- CLI commands: `cloudenv up`, `cloudenv status` (modified behavior)
- No web pages (CLI-only tool)

## Key Interactions to Verify
- `cloudenv up` with a multi-service docker-compose.yml creates ONE Fly Machine with all services as containers
- Services can communicate via service names (db, redis) resolving to localhost
- `cloudenv status` shows per-container health within the single machine
- `cloudenv down` still tears down correctly (app-scoped, not machine-scoped)

## Edge Cases
- docker-compose.yml with port conflicts between services (should error at parse time)
- docker-compose.yml with depends_on conditions (service_started, service_healthy)
- Single-service compose (should still use multi-container path, just 1 container)
- Service with `build:` and no Docker available (should error with clear message)
- /etc/hosts injection with service names containing special characters

## Critical Paths
- Full lifecycle: up (multi-container) → status → list → down
- Port conflict detection prevents silent failures at deploy time
- /etc/hosts injection makes service names resolve correctly across containers
