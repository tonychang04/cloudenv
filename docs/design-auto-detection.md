## Problem Statement

CloudEnv requires a docker-compose.yml to deploy. Most repos don't have one. A developer trying cloudenv for the first time has to write a compose file before they can see the product work. That's friction at the exact moment you need zero friction.

The goal: `cloudenv up` in any project directory, no config needed, returns a URL.

## Demand Evidence

- Founder personally wants to use cloudenv on repos that don't have docker-compose.yml
- The "first deploy" experience is the most critical moment for CLI tool adoption. Every extra step loses users.
- Heroku's `git push heroku main` and Railway's `railway up` proved that zero-config deploy is a killer feature

## Status Quo

Current cloudenv (v0.1.0, shipped and live-tested):
- Requires docker-compose.yml with `image:` fields
- Multi-container Machines: all services in one VM, shared localhost, /etc/hosts injection
- 78 tests, integration tested with nginx + postgres + redis on Fly.io

Users without docker-compose.yml must write one manually before deploying. This is the blocker.

## Constraints

- Detection must be invisible: user just runs `cloudenv up`, no flags needed
- Detection order: docker-compose.yml > Dockerfile > auto-detect stack (first match wins)
- Dockerfile generation uses opinionated defaults (80% path, not every edge case)
- Smart compose generation is heuristic-based, will sometimes be wrong
- Escape hatch: user adds a docker-compose.yml manually, existing path takes over
- Docker must be installed for auto-detection (need to build images)
- Generated `docker-compose.cloudenv.yml` is written to disk so user can inspect/edit

## Premises

1. Auto-detection should be invisible. No flags, no config, no prompts.
2. Dockerfile generation uses opinionated defaults (node:20-alpine, python:3.12-slim). Users who need custom builds add their own Dockerfile.
3. Smart compose generation (prisma → postgres, redis import → redis) is heuristic-based. When wrong, user adds a docker-compose.yml and existing path takes over.
4. Docker must be installed on the user's machine for auto-detection (build + push to Fly registry).
5. Detection order: docker-compose.yml > Dockerfile > auto-detect. First match wins.

## Approaches Considered

### Approach A: Static Detection, In-Memory Only
Scan for known files, pick a Dockerfile template, generate config in memory, deploy. User can't see or edit what was generated.

- Effort: M
- Pros: Simple, fast
- Cons: Opaque, user can't fix wrong guesses

### Approach B: LLM-Powered Detection
Send project structure to an LLM, generate Dockerfile and compose. Handles any stack.

- Effort: L
- Pros: Handles any stack
- Cons: Non-deterministic, requires API key, slow, hard to test

### Approach C: Static Detection + Write Compose to Disk (CHOSEN)
Detect stack, generate docker-compose.yml, write to project, then deploy using existing compose path. User can inspect and edit.

- Effort: M (human: ~1 week / CC: ~30 min)
- Pros: Transparent ("show your work"), editable, deterministic after first run
- Cons: Writes a file the user didn't ask for (mitigated by clear CLI output)

## Recommended Approach

**Approach C: Static Detection + Write Compose to Disk.**

Ship static detection now. Design LLM-powered detection as a future extension for stacks the static detector can't handle.

### Detection Flow

```
cloudenv up
  │
  ├── docker-compose.yml exists?
  │   └── YES → existing path (parse, deploy, done)
  │
  ├── Dockerfile exists?
  │   └── YES → generate single-service compose:
  │             services:
  │               app:
  │                 build: .
  │                 ports: [detected or 3000]
  │             Write to docker-compose.cloudenv.yml
  │             Build image, deploy
  │
  ├── Detect stack?
  │   ├── package.json → Node.js detector
  │   ├── requirements.txt / pyproject.toml → Python detector
  │   ├── go.mod → Go detector
  │   ├── Cargo.toml → Rust detector (deferred)
  │   ├── Gemfile → Ruby detector (deferred)
  │   └── NO MATCH → error: "Can't detect stack. Add a Dockerfile."
  │
  └── Stack detected:
      ├── Generate Dockerfile
      ├── Scan for DB dependencies:
      │   ├── prisma / pg / mysql2 → postgres service
      │   ├── redis / ioredis → redis service
      │   ├── mongoose / mongodb → mongo service
      │   └── (none) → single service only
      ├── Generate docker-compose.cloudenv.yml
      ├── Print: "Detected: Node.js + Postgres + Redis"
      ├── Print: "Generated docker-compose.cloudenv.yml"
      └── Build, deploy using existing compose path
```

### Stack Detectors

Each detector is a pure function: `(projectDir: string) => DetectedStack | null`

```typescript
interface DetectedStack {
  runtime: "node" | "python" | "go" | "rust" | "ruby";
  version: string;           // e.g., "20" for node, "3.12" for python
  entrypoint: string;        // e.g., "npm start", "python app.py"
  port: number;              // e.g., 3000, 8000, 8080
  databases: DetectedDB[];   // inferred from dependencies
}

interface DetectedDB {
  type: "postgres" | "redis" | "mongo" | "mysql";
  image: string;             // e.g., "postgres:16-alpine"
  port: number;              // e.g., 5432
  envVars: Record<string, string>;  // e.g., POSTGRES_PASSWORD: "cloudenv"
}
```

#### Node.js Detector

1. Read `package.json`
2. Version: check `engines.node` or default to "20"
3. Entrypoint: `scripts.start` or `scripts.dev` or "node index.js"
4. Port: search for `PORT` in code, check common frameworks (next → 3000, express → 3000, fastify → 3000)
5. Dependencies scan:
   - `prisma`, `@prisma/client`, `pg`, `postgres` → postgres
   - `redis`, `ioredis`, `@redis/client` → redis
   - `mongoose`, `mongodb` → mongo
   - `mysql2`, `mysql` → mysql

#### Python Detector

1. Read `requirements.txt` or `pyproject.toml`
2. Version: default "3.12"
3. Entrypoint: detect framework:
   - `django` → "python manage.py runserver 0.0.0.0:8000"
   - `flask` → "flask run --host=0.0.0.0"
   - `fastapi` → "uvicorn main:app --host 0.0.0.0 --port 8000"
   - fallback → "python app.py"
4. Port: 8000 (django/fastapi) or 5000 (flask)
5. Dependencies: `psycopg2`, `asyncpg` → postgres, `redis` → redis, `pymongo` → mongo

#### Go Detector

1. Read `go.mod`
2. Version: parse from `go.mod`
3. Entrypoint: "go run ." or detect main package
4. Port: 8080 (Go convention)
5. Dependencies: `pgx`, `pq` → postgres, `go-redis` → redis

#### Dockerfile Templates

Each generates a multi-stage Dockerfile:

**Node.js:**
```dockerfile
FROM node:{version}-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build 2>/dev/null || true

FROM node:{version}-alpine
WORKDIR /app
COPY --from=builder /app .
EXPOSE {port}
CMD {entrypoint}
```

**Python:**
```dockerfile
FROM python:{version}-slim
WORKDIR /app
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
EXPOSE {port}
CMD {entrypoint}
```

### Generated Compose File

Written to `docker-compose.cloudenv.yml` (not `docker-compose.yml` to avoid overwriting user's existing file if they add one later).

For a Node.js project with Prisma (postgres) and redis:

```yaml
# Generated by cloudenv auto-detection
# Edit this file to customize your environment
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://postgres:cloudenv@db:5432/app
      REDIS_URL: redis://redis:6379
    depends_on:
      - db
      - redis
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: cloudenv
      POSTGRES_DB: app
  redis:
    image: redis:7-alpine
```

### Future: LLM-Powered Detection

When static detection fails (unknown stack, complex monorepo, custom build system), offer an LLM fallback:

```
$ cloudenv up
Can't auto-detect this project's stack.

Options:
  1. Add a Dockerfile or docker-compose.yml manually
  2. Run `cloudenv up --ai` to let AI analyze your project (requires API key)
```

The `--ai` flag would:
1. Collect project structure (file tree, package files, entry points)
2. Send to Claude/GPT with a prompt: "Generate a Dockerfile and docker-compose.yml for this project"
3. Write the generated files to disk
4. Deploy using the existing compose path

This is deferred. Ship static detection first, add LLM when we hit stacks the detector can't handle.

## Resolved Questions

1. **File naming:** Generate `docker-compose.cloudenv.yml`. Update the detection chain in `up.ts` to check for files in this order: `docker-compose.yml` → `docker-compose.cloudenv.yml` → auto-detect. The `-f` flag overrides both.
2. **Port detection:** Each detector has a hardcoded default port (Node: 3000, Python: 8000, Go: 8080). Source code scanning is attempted first, but the default is always the fallback. Never fail because of port detection.
3. **.dockerignore:** Always generate `.dockerignore` alongside the Dockerfile if one doesn't exist. Include: `node_modules`, `.git`, `dist`, `__pycache__`, `.env`, `.fly`. This is a required step, not optional.
4. **Rust/Ruby detectors:** Deferred to follow-up. Ship Node.js, Python, and Go first (covers the vast majority of projects). Rust and Ruby detectors are listed in the flow diagram but NOT in scope for this implementation.

## Success Criteria

1. `cloudenv up` in a Node.js project with no docker-compose.yml generates a Dockerfile, compose file, and deploys successfully
2. Detection correctly identifies postgres/redis/mongo dependencies from package.json
3. Generated compose file is valid and can be edited by the user
4. Subsequent `cloudenv up` uses the generated file (no re-detection)
5. `cloudenv up` in a project with an existing Dockerfile uses it directly

## What I noticed about how you think

- "First have static, but also write out another path for LLM-powered detection." You're building in layers. Ship the deterministic thing, keep the door open for the smarter thing. That's exactly right.
- You went straight to the most ambitious option (B+C+smart compose). You're not afraid of scope when the payoff is clear. The zero-config deploy experience is worth the investment.
- You keep coming back to "does it work from the user's perspective." Not "is the architecture clean" or "is the code elegant." You're optimizing for the person running `cloudenv up` for the first time.
