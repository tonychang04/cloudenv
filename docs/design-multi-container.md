## Problem Statement

CloudEnv V1 creates one Fly Machine per docker-compose service. A 3-service stack (web + db + redis) means 3 separate VMs on Fly's private network. This is expensive (~3x cost), slow to provision (sequential machine creation), and adds networking complexity (inter-service traffic crosses VM boundaries). For ephemeral preview environments that spin up and tear down constantly, the per-machine overhead is the bottleneck.

## Demand Evidence

- Founder identified cost, networking simplicity, and boot speed as equally important drivers
- Original design doc targets 2-4 second provisioning for warm pools. Current multi-machine approach can't hit this because each machine provisions sequentially
- Target users run 5-10 simultaneous environments. At 3 machines per env, that's 15-30 machines vs 5-10 with consolidation. Direct cost impact.

## Status Quo

Current CloudEnv V1 (just shipped, 63 tests passing, live integration tested):
- Reads docker-compose.yml → creates N Fly Machines (one per service)
- Each machine runs a single Docker image
- Inter-service networking via Fly's private network (fdaa:: addresses)
- Works but expensive, slow, and complex for the ephemeral use case

## Target User & Narrowest Wedge

Same as V1: founder/CTO running multiple AI coding sessions who needs isolated full-stack environments per branch. The wedge narrows further: `cloudenv up` should feel as fast and cheap as running docker-compose locally, but give you a public URL.

## Constraints

- Must work with unmodified docker-compose.yml files (no user-side changes)
- Service names (db, redis, web) must resolve correctly between containers
- Environment provisioning target: under 10 seconds for a 3-service stack (stretch: under 5 seconds)
- Must support services with `image:` fields (pre-built images). `build:` support is stretch goal (requires Docker on user's machine to build + push to registry first).
- Single Fly Machine per environment (all services co-located)

## Premises

1. Fly.io multi-container Machines are stable enough for production use (documented, not labeled experimental)
2. All docker-compose services can run in a single Fly Machine sharing localhost, including databases like Postgres and Redis
3. The `containers` array in the Machines API maps to docker-compose services: each service becomes a container entry with its own image, env vars, healthchecks, and depends_on
4. Inter-service networking works via /etc/hosts injection: service names (db, redis) resolve to 127.0.0.1. Connection strings like `postgres://db:5432/myapp` work unmodified because all containers share the same network namespace
5. One machine = one failure domain. If the VM dies, all services die. For ephemeral preview environments, this is acceptable.

## Approaches Considered

### Approach A: Multi-Container Machines (native Fly API, rewrite env vars)
Map each docker-compose service to a `containers` array entry. Rewrite environment variables that reference service names (e.g., DATABASE_URL from `postgres://db:5432` to `postgres://localhost:5432`).

- Effort: M (human: ~1 week / CC: ~30 min)
- Risk: Low
- Pros: Cheapest, fastest boot, simplest networking
- Cons: Must detect and rewrite connection strings (fragile, may miss some)

### Approach B: s6-overlay Single Image
Generate a Dockerfile at runtime that bundles all services into one image using s6-overlay as process supervisor. Build, push, run as a single container.

- Effort: L (human: ~2 weeks / CC: ~45 min)
- Risk: Medium
- Pros: Works on any container platform
- Cons: Requires Docker for every deploy, slow image builds, reinvents what Fly already built

### Approach C: Multi-Container with /etc/hosts injection (CHOSEN)
Same as A, but inject /etc/hosts entries (`db → 127.0.0.1`, `redis → 127.0.0.1`) into each container via Fly's `files` API. Connection strings stay untouched. Docker-compose.yml works as-is.

- Effort: M (human: ~1 week / CC: ~30 min)
- Risk: Low
- Pros: Zero friction, no env var rewriting, docker-compose works unmodified, same cost/speed benefits
- Cons: Relies on /etc/hosts being respected (standard behavior for all major runtimes)

## Recommended Approach

**Approach C: Multi-Container Machines with /etc/hosts injection.**

### How It Works

```

┌─────────────────┐        ┌────────────────────────────────────┐

│     image: nginx │        │   "config": {                      │
│     ports: 80    │        │     "containers": [                │
│   db:            │        │       { "name": "web",             │
│     image: pg:16 │        │         "image": "nginx",          │
│   redis:         │        │         "files": [/etc/hosts],     │
│     image: redis │        │         "healthchecks": [...] },   │
└─────────────────┘        │       { "name": "db",              │
                           │         "image": "postgres:16",     │

```

### /etc/hosts Content

For a docker-compose.yml with services `web`, `db`, `redis`:

```
127.0.0.1 localhost
::1 localhost
127.0.0.1 web
127.0.0.1 db
127.0.0.1 redis
```

Injected into every container via Fly's `files` API:
```json
{
  "files": [{
    "guest_path": "/etc/hosts",
    "raw_value": "MTI3LjAuMC4xIGxvY2FsaG9zdAoxMjcuMC4wLjEgd2ViCjEyNy4wLjAuMSBkYgoxMjcuMC4wLjEgcmVkaXMK"
  }]
}
```
(base64-encoded /etc/hosts content)

### Container Configuration Mapping

| docker-compose field | Fly containers field |
|---------------------|---------------------|
| `image:` | `image` |
| `environment:` | `env` |
| `command:` | `cmd` |
| `depends_on:` | `depends_on` (with `condition: "started"` or `"healthy"`) |
| `ports:` (first web service) | top-level `services` on the Machine config |
| `healthcheck:` | `healthchecks` array (HTTP, TCP, or exec) |

### NOT in scope (deferred)

- **Auto-detection of Dockerfile-only or no-config projects.** Future feature. This design only covers docker-compose.yml input.
- **`build:` support.** Requires Docker on user machine. Deferred until demand is validated.
- **Volume mounts.** Ephemeral environments don't persist data across deploys.
- **Custom resource sizing flags** (`--cpu`, `--memory`). Ship with sensible defaults first.

## Migration from V1

The multi-container approach replaces the per-service machine creation loop in `up.ts`. Both paths share the same `down.ts` teardown logic (destroy all machines in the app, then delete the app), so teardown requires no changes. The env-store format is unchanged, just `services` array entries now reference containers within one machine instead of separate machines.

V1 code is not feature-flagged. The multi-container path replaces it directly. Existing environments created with V1 can still be destroyed with `cloudenv down` because teardown is app-scoped, not machine-scoped.

**Env-store changes:** The `EnvRecord.services` array currently stores one `machineId` per service. With multi-container, all services share one machine. The `machineId` field becomes the same value for all services in an environment. The `down` command already destroys by app name (not by individual machine IDs), so no teardown logic changes. The `status` command calls `listMachines()` which returns one machine. Service-level status comes from the machine's container states, not from separate machine states. This may require a minor update to `status.ts` to parse container-level health from the single machine response.

## Unsupported Compose Features

When parsing docker-compose.yml, warn and skip these unsupported fields (existing behavior from V1, no changes needed):
- `volumes:` — not supported in ephemeral environments
- `networks:` — replaced by shared localhost networking
- `secrets:`, `configs:` — not supported
- `deploy:` — Fly handles resource allocation
- `build:` — stretch goal, requires Docker on user machine. Error with clear message if no `image:` field is present.

## Resolved Questions

1. **Guest resources:** Default to `shared-cpu-2x` with `2048MB` RAM. Postgres alone wants 256MB shared_buffers, Redis needs headroom, and the web server adds more. 2GB handles realistic preview workloads. Users can override with `--cpu` and `--memory` flags (future).
2. **Port conflicts:** Detect at parse time. If two services expose the same port (via `ports:` or `expose:`), error with: "Services X and Y both listen on port N. This is not supported in single-machine mode." Also detect well-known default ports (5432 for postgres images, 6379 for redis images, 3306 for mysql images) even when not explicitly declared, since these services bind to their default ports on startup. This is a parse-time check in compose.ts.
3. **Volumes:** Deferred. Same as V1. Warn and skip.
4. **Build context:** Services with `build:` and no `image:` error with: "Service X uses build: which requires Docker. Add an image: field or install Docker."
5. **depends_on mapping:** docker-compose `service_started` maps to Fly's `started`, `service_healthy` maps to `healthy`. `service_completed_successfully` maps to `exited_successfully`. If a compose service has no condition specified, default to `started`.
6. **Web service detection:** The first service with a `ports:` field is the web service. Its first exposed port becomes the Machine-level `services` entry (HTTP/HTTPS routing). If multiple services expose ports, only the first gets public routing. Others are internal-only.
7. **/etc/hosts injection:** Fly's `files` API replaces the file content entirely. The generated /etc/hosts includes standard entries (`127.0.0.1 localhost`, `::1 localhost`) plus all service names mapped to both IPv4 and IPv6 loopback (e.g., `127.0.0.1 db` and `::1 db`). This prevents IPv6-first resolvers from timing out before falling back to IPv4. **VALIDATION REQUIRED:** Before implementing, run a spike to confirm that (a) the `files` API works per-container in multi-container mode, and (b) the container runtime does not regenerate /etc/hosts after injection.

## Spike: Validate /etc/hosts injection

Before implementing the full feature, run this manual test:

1. Create a Fly app with a 2-container Machine via the API
2. Inject a custom /etc/hosts file into both containers using the `files` field
3. Exec into each container and verify /etc/hosts contains the injected content
4. From container A, verify that the service name of container B resolves to 127.0.0.1
5. From container A, connect to a port served by container B using the service name

If this works, the approach is validated. If Fly regenerates /etc/hosts, fall back to Approach A (env var rewriting).

## Success Criteria

1. `cloudenv up` with a 3-service docker-compose.yml (web + db + redis) creates a single Fly Machine with all services running
2. Services can communicate using their docker-compose names (e.g., web connects to `db:5432`)
3. Public URL serves traffic from the web service
4. Provisioning time under 10 seconds (stretch: under 5 seconds)
5. Cost is ~1/3 of current multi-machine approach
