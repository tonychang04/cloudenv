# Deploy to Ephemeral Preview Environment

You are deploying this project to Fly.io using the `cloudenv` CLI. Your job is to analyze the project, fix any issues that would prevent cloud deployment, and call `cloudenv up` to get a working URL.

## Prerequisites

- `cloudenv` CLI must be installed (`npm install -g cloudenv`)
- User must be logged in (`cloudenv login --token <fly-api-token>`)
- `flyctl` installed (preferred, builds remotely on Fly's hardware, fast) OR Docker running (fallback, builds locally, slower on Apple Silicon)

Check prerequisites:
```bash
which cloudenv || echo "MISSING: run npm install -g cloudenv"
cloudenv list 2>&1 | head -1 || echo "MISSING: run cloudenv login"
which flyctl && echo "flyctl: OK (remote builds, fast)" || echo "flyctl: not found"
docker info >/dev/null 2>&1 && echo "docker: OK (local builds, fallback)" || echo "docker: not running"
# Need at least one of flyctl or docker for services with build:
```

**Build path selection (automatic):**
- `flyctl` installed → builds on Fly's amd64 machines (~30s, no QEMU)
- Only Docker → builds locally with `--platform linux/amd64` (slow on Apple Silicon, ~5-10min first time)
- Neither → errors if any service needs building. Services with `image:` still work.

## Step 1: Find the Right Compose File

Look for compose files in this order. Use the first one found:

1. `docker-compose.prod.yml` or `docker-compose.production.yml` (preferred for cloud)
2. `docker-compose.yml` or `docker-compose.yaml`
3. `docker-compose.cloudenv.yml` (generated by cloudenv auto-detect)

If none exist, check if cloudenv can auto-detect the stack:
- `package.json` → Node.js
- `requirements.txt` → Python
- `go.mod` → Go
- `Dockerfile` → direct build

If auto-detection works, `cloudenv up` handles it. Skip to Step 5.

**Dev vs Prod:** If only a dev compose file exists (look for `target: dev`, heavy volume mounts of source code, `npm install` in command), you'll need to adapt it. Prefer prod if available.

## Step 2: Analyze for Cloud-Readiness

Read the compose file and check each service for these issues:

### Bind Mounts (files from local filesystem)
Look for volumes like:
- `./deploy/init.sql:/docker-entrypoint-initdb.d/init.sql` — local file mounted into container
- `./src:/app/src` — source code mounted (dev pattern)
- `./config.yml:/etc/config.yml` — config file mounted

**For each bind mount:**
1. Read the source file to understand what it does
2. Determine if the service will fail without it (init SQL = critical, log config = optional)
3. If critical: generate a wrapper Dockerfile (Step 3)
4. If optional: note it but don't block deployment

### Environment Variables
Look for `${VAR:-default}` patterns. cloudenv resolves these to defaults automatically.

For env vars WITHOUT defaults (`${VAR}` or `$VAR`):
- Check if a `.env` file exists in the project root
- If yes, read values from it
- If no, ask the user what values to use for critical vars (DB passwords, JWT secrets)
- For preview environments, generate random secrets: `openssl rand -hex 32`

### Docker Socket
Services mounting `/var/run/docker.sock` CANNOT work on Fly. Skip them with `--skip` or remove from compose.

### Build Targets
If a service uses `target: dev`, check if a `target: runner` or `target: production` exists in the same Dockerfile. Prefer the production target.

## Step 3: Generate Wrapper Dockerfiles

For services that have critical bind mounts, generate a wrapper Dockerfile that bakes the files in.

**Example:** postgres service mounts init SQL files:
```yaml
# Original compose
postgres:
  image: ghcr.io/myapp/postgres:v15
  volumes:
    - ./deploy/db/init.sql:/docker-entrypoint-initdb.d/01-init.sql
    - ./deploy/db/config.conf:/etc/postgresql/postgresql.conf
```

**Generate** `Dockerfile.cloudenv.postgres`:
```dockerfile
FROM ghcr.io/myapp/postgres:v15
COPY deploy/db/init.sql /docker-entrypoint-initdb.d/01-init.sql
COPY deploy/db/config.conf /etc/postgresql/postgresql.conf
```

**Example:** deno service mounts function source code:
```yaml
deno:
  image: denoland/deno:alpine-2.0.6
  volumes:
    - ./functions:/app/functions
  command: deno run --allow-net functions/server.ts
```

**Generate** `Dockerfile.cloudenv.deno`:
```dockerfile
FROM denoland/deno:alpine-2.0.6
WORKDIR /app
COPY functions/ /app/functions/
```

Write these files to the project directory. They are temporary build artifacts.

## Step 4: Generate Cloud-Ready Compose File

Write `docker-compose.cloudenv.yml` that:
- Replaces `image:` with `build:` pointing to wrapper Dockerfiles for adapted services
- Keeps `image:` for services that don't need adaptation
- Removes bind mount volumes
- Removes network definitions (cloudenv uses shared localhost)
- Removes docker.sock services entirely
- Resolves env vars to concrete values
- Uses production build targets where available

**Example output:**
```yaml
# Generated by cloudenv skill — do not edit
services:
  postgres:
    build:
      context: .
      dockerfile: Dockerfile.cloudenv.postgres
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: myapp
    ports:
      - "5432:5432"

  postgrest:
    image: postgrest/postgrest:v12.2.12
    environment:
      PGRST_DB_URI: postgres://postgres:postgres@postgres:5432/myapp
      PGRST_DB_SCHEMA: public
      PGRST_DB_ANON_ROLE: anon
      PGRST_JWT_SECRET: dev-secret-please-change-in-production
    ports:
      - "5430:3000"
    depends_on:
      - postgres

  app:
    build:
      context: .
      dockerfile: Dockerfile
      target: runner
    ports:
      - "7130:7130"
    environment:
      DATABASE_URL: postgresql://postgres:postgres@postgres:5432/myapp
      JWT_SECRET: dev-secret-please-change-in-production
    depends_on:
      - postgres
```

## Step 5: Deploy

Run cloudenv:
```bash
cloudenv up -f docker-compose.cloudenv.yml
```

Or if no adaptation was needed:
```bash
cloudenv up
```

If a specific port should be the public-facing one:
```bash
cloudenv up --port 7130
```

## Step 6: Verify

After deployment succeeds:
1. Wait 10 seconds for services to fully start
2. Check status: `cloudenv status`
3. Curl the URL: `curl -s <url> | head -20`
4. Report the result to the user

If the URL isn't responding:
1. Check container health: `cloudenv status --json`
2. If services are unhealthy, check which one and why
3. Common fixes: wrong port, missing env var, DB not initialized

## Cleanup

Generated files to add to `.gitignore`:
```
Dockerfile.cloudenv.*
docker-compose.cloudenv.yml
```

To tear down: `cloudenv down` or `cloudenv down <env-name> --force`

## Common Patterns

### Node.js + Postgres (e.g., Next.js, Express)
- Web service on port 3000 or specified in package.json scripts
- Postgres with init SQL for schema
- Usually needs wrapper Dockerfile for postgres init files

### Python + Postgres (e.g., Django, FastAPI)
- Web service on port 8000 (django) or 8000 (fastapi)
- May need migrations run at startup
- Add migration command to the compose command

### Multi-service apps (e.g., InsForge)
- Multiple services with interdependencies
- Some services need source code baked in (not mounted)
- Some services (log shippers, monitoring) can be skipped
- Use `--port` to specify the main web service port
